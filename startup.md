# Startup OS (Solo CEO + AI Co-Founding Teams)

本文件是给“一个人公司 CEO”的操作手册：你不组建真人团队，而是把各种 AI 当作联创与部门负责人来编排，形成可持续的产品迭代节奏。

本仓库相关文档入口：

- `business_plan.md`：商业与发行策略总览
- `replay.md`：本地回放（`HandSpec -> ReplayTape`）最小技术方案
- `rei.md`：REI 解说系统（Lite/Online、触发、降级、credits）
- `agent.md`：HandSpecBuilder / Director / BotPolicy 等“会行动的 Agent”

---

## 0. 核心原则（别被 AI 带跑偏）

- **Artifact-first**：先产出“可被审阅的工件”，再写代码/做内容。工件包括 PRD、设计稿、数据结构、测试计划、商店页文案等。
- **Hard truth**：游戏规则/引擎/协议是事实来源，AI 只能提案不能拍板；任何“看似合理”都必须能被引擎/测试验证。
- **Short by default**：默认短输出。长文只在你明确要求“深度复盘/深度分析”时出现。
- **成本可控**：把 LLM 当作 COGS（变动成本）。没有额度/缓存/降级，就等着亏。
- **单人节奏**：你能持续的迭代速度，比任何“宏大规划”更重要。

## 1. 你的“AI 联创团队”编制（MVP）

你是 CEO，同时也是最终的“编剧/审稿/合并者”。AI 团队是你的虚拟部门负责人。

建议的 AI 角色（按调用频率从高到低）：

### 1.1 产品负责人（CPO Agent）

职责：

- 把“想做什么”收敛成可交付的 MVP 切片
- 写 PRD：用户故事、成功指标、边界、非目标、验收标准

你给它的输入：

- 当前里程碑、约束（时间、平台、法律边界、预算）
- 相关文档：`business_plan.md`、`rei.md`、`replay.md`、`agent.md`
- 当前产品截图/录屏（关键）

它必须交付：

- 1 页 PRD（带验收标准）
- 风险清单（产品/合规/成本）
- 砍需求建议（必须给出砍掉的理由）

### 1.2 技术负责人（CTO/Architect Agent）

职责：

- 把 PRD 落成最小技术方案：数据结构、接口、状态机、存储、错误模型
- 明确不做/延后做的部分（避免工程泥潭）

必须交付：

- 设计文档（模块、数据流、边界、失败模式）
- 实施步骤（3–6 步）
- 可验证清单（如何证明“做对了”）

### 1.3 实现工程师（Implementer Agent）

职责：

- 直接改仓库代码、跑测试、做最小可用实现

必须交付：

- 可运行的 patch
- 最小验证步骤（命令+预期结果）

### 1.4 代码审阅（Reviewer Agent）

职责：

- 专挑 bug、回归、边界条件、并发与状态一致性问题
- 要求缺失的测试/日志/错误处理

必须交付：

- 按严重性排序的问题清单
- 必须改 vs 可稍后改

### 1.5 QA / 测试设计（QA Agent）

职责：

- 针对“回放/分支/解说触发”等高风险交互写测试策略与手工脚本

必须交付：

- 冒烟用例（5–10 条）
- 回归用例（覆盖关键路径）
- 失败复现模板（便于你以后追 bug）

### 1.6 视觉与体验（Design Agent）

职责：

- 保持 cyberpunk 视觉一致性：排版、动效、信息层级
- 设计“少打扰”的 REI 面板与时间线控制

必须交付：

- 信息层级规则（哪些永远短、哪些要折叠）
- 组件规格（间距、字号、状态、动效节奏）

### 1.7 文案与叙事（Copy/Narrative Agent）

职责：

- Steam 商店页、更新日志、游戏内微文案（REI Lite 模板）

必须交付：

- 文案分层：短句（默认）/长句（展开）/系统状态（联网、降级、错误）

### 1.8 数据与增长（Analyst/Growth Agent）

职责：

- 定义埋点与指标：留存、复盘使用率、订阅转化、credits 消耗

必须交付：

- 指标字典（每个指标怎么计算）
- 1–2 个能立刻指导迭代的实验建议

### 1.9 合规与风控（Legal/Policy Agent）

职责：

- 帮你把产品边界写成 ToS/提示语/禁用行为与执行策略（不替代律师）

必须交付：

- 风险点清单（按地区/平台）
- 建议的 UI 告知与条款条目

## 2. 单人公司工作流（从想法到上线）

### 2.1 最小闭环

1. CEO：一句话目标（本周）
2. CPO：PRD（1 页）
3. CTO：技术方案（模块+数据结构+失败模式）
4. Implementer：落地 patch + 最小验证
5. Reviewer + QA：找 bug + 回归脚本
6. CEO：合并与发布说明（Steam 更新日志/路线图）

### 2.2 你该如何“喂上下文”

每次调用 AI，固定提供：

- 目标：一句话 + 目标日期
- 约束：平台、性能、成本、合规边界
- 事实：相关文档链接（本仓库 markdown）、代码路径、截图
- 输出格式：要求结构化（标题/列表/JSON/schema），不要散文

强制 AI 写清楚：

- Assumptions（它假设了什么）
- Open questions（它缺什么信息）
- Out of scope（它明确不做什么）

## 3. 工件模板（复制即用）

### 3.1 PRD（产品需求文档）

- 背景：为什么做
- 目标用户：谁在用、场景是什么
- 核心路径：用户从 A 到 B 的最短路径
- 需求列表（MVP）：只写 3–7 条
- 非目标：明确不做（至少 3 条）
- 验收标准：可观察、可测试
- 风险：成本/合规/体验/技术

### 3.2 Tech Spec（技术方案）

- 范围：包含/不包含
- 数据结构：输入/输出 schema（最好有 JSON 示例）
- 模块边界：谁拥有状态，谁只消费事件
- 失败模式：非法输入、网络断开、缓存击穿、性能退化
- 验证：怎么证明正确（测试/日志/指标）

### 3.3 QA Test Plan（测试计划）

- 冒烟（5–10 条）
- 回归（覆盖关键路径）
- 破坏性用例（非法 action、重复卡、seek/branch 边界）
- 观测点（需要哪些日志/指标）

### 3.4 Release Notes（更新说明）

- 新增：对用户可见的 2–5 条
- 修复：1–3 条
- 已知问题：1–3 条（避免差评爆炸）
- 下一步：1–2 条（别写宏大叙事）

## 4. “AI 功能”成本护栏（从第一天就要有）

即使你今天不接 LLM，也要按未来会接来设计：

- 额度（subscription 内含 credits）：永不做无限
- 降级：credits 归零或网络不可用 -> REI Lite
- 缓存：同一 `HandSpec + step` 讲解要命中缓存
- 输出限制：默认 1 行 + 0–3 行详情（展开才长）
- 风控：限速、防刷、去重、最大上下文长度

这些在 `rei.md` 和 `business_plan.md` 已有更细拆分，本文件只强调“操作上必须落地”。

---

## 5. 迭代节奏（个人项目最稳的节拍）

建议的周节拍（示例）：

- 周一：PRD + Tech Spec（只做一件大事）
- 周二/三：实现 + 自测
- 周四：打磨体验 + QA 回归
- 周五：发布/录屏/写更新日志/收反馈

每日节拍（15–30 分钟）：

- 写下今天唯一目标（可交付物）
- 先跑最小验证，再继续扩

## 6. 建议你先打穿的 2 个“个人项目护城河”

这两件做成了，后续再加 LLM/订阅才不虚：

1. **回放时间线体验**：step/back/seek/branch 足够顺滑（见 `replay.md`）
2. **REI Lite 可信度**：模板也要“强绑定局面”，不做泛泛的占位状态（见 `rei.md`）

## 7. Prompt 配方（给 AI 的“岗位说明书”）

### 7.1 CPO Agent Prompt

- 你是产品负责人。目标：把下面目标拆成 1 页 PRD。
- 输出必须包含：MVP 切片、非目标、验收标准、风险与砍需求建议。
- 禁止：泛泛而谈、写成愿景文章。

### 7.2 CTO Agent Prompt

- 你是技术负责人。基于 PRD 产出 Tech Spec。
- 输出必须包含：数据结构示例、模块边界、失败模式、验证清单、实施步骤。

### 7.3 Reviewer Agent Prompt

- 你是严苛 code reviewer。只找问题，按严重性排序。
- 强制：指出缺失测试与回归风险。

### 7.4 QA Agent Prompt

- 你是 QA。给出 10 条以内冒烟 + 15 条以内回归。
- 强制覆盖：回放 seek/back、分支、非法输入、降级路径。

---

## 8. 你作为 CEO 的“最终职责”

AI 可以把你推向速度，也可以把你推向幻觉。你最终必须做到：

- 决策收敛（不做“也许都要”）
- 对质量负责（bug/作弊/合规/差评最终都是你的）
- 对现金流负责（LLM 成本、平台抽成、运营成本）

把“能持续发布”当作第一 KPI。
